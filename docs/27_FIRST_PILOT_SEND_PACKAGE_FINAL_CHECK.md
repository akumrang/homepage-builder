## 1. 목적

이 문서는 아직 실제 파일럿 고객이 정해지지 않은 상태에서, 향후 첫 파일럿 고객이 생겼을 때 사용할 자료 요청 패키지가 내부 기준으로 준비되었는지 점검한 기록이다.

현재 단계는 실제 고객 발송이 아니다. 고객명, 연락처, 회신 기한, 제출 링크 같은 실고객 정보는 정해지지 않았고, 이 저장소에도 기록하지 않는다.

검토 대상:

- `docs/17_FIRST_PILOT_CUSTOMER_SEND_COPY.md`
- `docs/14_FIRST_PILOT_CUSTOMER_INTAKE_PACKET.md`
- `docs/16_FIRST_PILOT_INTAKE_PACKET_FINAL_REVIEW.md`

## 2. 최종 판정

판정: `READY FOR VIRTUAL REHEARSAL`

고객 발송 자료의 구조와 문구는 첫 파일럿 준비 기준으로 사용 가능하다.

다만 실제 고객이 생기기 전에는 아래 값을 채우지 않는다. 실제 파일럿 고객이 정해진 뒤에만 묵산이 저장소 밖에서 다음 값을 채운다.

- 고객 학원명
- 담당자 호칭
- 회신 기한
- 파일 제출 방식
- 샘플 홈페이지 링크
- 베틀 시스템 문의 연락처

실제 고객명, 개인 연락처, 이메일, 고객별 제출 링크는 어떤 경우에도 저장소에 커밋하지 않는다.

## 4. 이번 점검에서 보완한 내용

`docs/17_FIRST_PILOT_CUSTOMER_SEND_COPY.md`에 다음을 추가했다.

1. 고객에게 보내는 범위와 내부 전용 범위를 분리했다.
2. 실제 고객 정보는 저장소에 기록하지 않는다는 원칙을 다시 명시했다.
3. 사진은 원본 그대로 보내도 되며, 베틀 시스템이 사용 가능 여부를 확인한다는 안내를 추가했다.
4. 내부 발송 체크에 “고객에게 보낼 범위만 복사했다” 항목을 추가했다.

## 5. 지금 묵산이 할 일

지금은 실제 발송을 승인하는 단계가 아니다.

묵산은 이 패키지가 향후 파일럿 고객에게 보낼 기본 양식으로 적정한지만 판단하면 된다.

확인할 것:

- 고객이 직접 홈페이지를 만드는 제작툴로 오해하지 않는가
- 베틀 시스템이 완성형 홈페이지를 제작한다는 구조가 분명한가
- 질문량이 첫 파일럿 기준으로 과하지 않은가
- 개인정보와 민감 자료를 보내지 말라는 기준이 충분한가
- 실제 고객이 생겼을 때 아래 값을 저장소 밖에서 채울 수 있는가

| 항목 | 작성 기준 |
|---|---|
| 고객 학원명 | 실제 학원명 |
| 담당자 호칭 | 원장님, 실장님, 담당자님 등 |
| 회신 기한 | 첫 파일럿은 3-5영업일 권장 |
| 파일 제출 방식 | 이메일 첨부, 드라이브 링크, 메신저 첨부 중 하나 |
| 샘플 홈페이지 링크 | 배포 URL 또는 내부 확인용 URL |
| 베틀 시스템 문의 연락처 | 실제 응답 가능한 연락 방식 |

실제 고객이 생긴 뒤 고객에게 보낼 때는 `docs/17_FIRST_PILOT_CUSTOMER_SEND_COPY.md`의 고객 발송 범위만 복사한다.

## 6. 실제 고객이 생겨도 발송하면 안 되는 경우

향후 실제 파일럿 고객이 생기더라도, 다음 중 하나라도 해당하면 발송을 보류한다.

- 실제 고객명이나 연락처를 저장소 문서에 직접 적어야 하는 상황
- 회신 기한 또는 파일 제출 방식이 아직 정해지지 않은 상황
- 샘플 홈페이지 링크가 아직 고객에게 보여줄 상태가 아닌 상황
- 문의를 받을 실제 연락 방식이 정해지지 않은 상황
- 고객에게 내부 발송 체크나 내부 공정 문구까지 함께 보낼 위험이 있는 상황
