Last Updated: 2026-06-30

## 1. 목적

이 문서는 실제 고객이 없는 상태에서 첫 파일럿 발송용 자료 요청 사본을 가상 정보로 채워보는 리허설 기록이다.

현재 단계는 실제 발송이 아니다. 이 문서의 모든 학원명, 담당자, 회신 기한, 제출 방식, 링크, 연락처는 가상값 또는 placeholder다.

검증 대상:

- `docs/17_FIRST_PILOT_CUSTOMER_SEND_COPY.md`
- `docs/27_FIRST_PILOT_SEND_PACKAGE_FINAL_CHECK.md`

---

## 2. 리허설 판정

판정: `PASS WITH NOTES`

첫 파일럿 발송 문안은 가상 정보를 넣었을 때 전체 흐름이 자연스럽다.

다만 실제 고객이 생기기 전에는 다음 값은 확정하지 않는다.

- 실제 고객 학원명
- 실제 담당자 호칭
- 실제 회신 기한
- 실제 파일 제출 방식
- 실제 샘플 홈페이지 링크
- 실제 문의 연락처

---

## 3. 가상 입력값

아래 값은 리허설용이다. 실제 고객 정보가 아니다.

| 항목 | 가상값 |
|---|---|
| 학원명 | `[가상] 온빛국어학원` |
| 담당자 호칭 | `원장님` |
| 회신 기한 | `2026-07-07` |
| 파일 제출 방식 | `[가상] 이메일 첨부 또는 내부 리허설용 드라이브 placeholder` |
| 베틀 시스템 문의 연락처 | `[가상 문의 연락처 placeholder]` |
| 샘플 홈페이지 링크 | `http://127.0.0.1:5175/h/sample-korean-academy` |

주의:

- 위 값은 실제 발송에 사용하지 않는다.
- 실제 고객이 생기더라도 이 저장소에는 고객 실명, 연락처, 실제 제출 링크를 기록하지 않는다.

## 4. 가상 고객 발송 본문

아래는 `docs/17_FIRST_PILOT_CUSTOMER_SEND_COPY.md`의 고객 발송 범위 중 본문만 가상값으로 채운 결과다.

```text
안녕하세요, [가상] 온빛국어학원 원장님.
베틀 시스템 홈페이지 초안 제작을 위해 자료 요청 양식을 보내드립니다.

이번 홈페이지는 고객님이 직접 편집하는 제작툴이 아니라, 베틀 시스템이 표준 템플릿 안에서 학원별 정보를 정리하여 완성형 홈페이지로 제작하는 방식입니다.

모든 항목을 완벽히 채우지 않으셔도 됩니다.
가능한 항목부터 보내주시면 베틀 시스템이 부족한 부분을 확인해 다시 요청드리겠습니다.

로고나 사진이 없더라도 우선 제작을 시작할 수 있습니다.
다만 공개 가능한 정보, 사진 사용 동의, 개인정보 미포함 여부는 게시 전 반드시 확인합니다.

회신 기한: 2026-07-07
파일 제출 방식: [가상] 이메일 첨부 또는 내부 리허설용 드라이브 placeholder
샘플 확인 링크: http://127.0.0.1:5175/h/sample-korean-academy

확인 중 궁금한 점이 있으면 [가상 문의 연락처 placeholder]로 편하게 말씀 주세요.
```

판정:

- 문장 흐름은 자연스럽다.
- `베틀 시스템` 외부 브랜드 표기는 일관된다.
- 고객이 직접 홈페이지를 편집하는 제작툴로 오해할 가능성은 낮다.
- 샘플 링크가 localhost이면 실제 고객에게는 사용할 수 없다. 실제 고객 발송 전에는 배포 URL이 필요하다.

## 5. 고객에게 보낼 범위 점검

`docs/17_FIRST_PILOT_CUSTOMER_SEND_COPY.md` 기준 고객에게 보내는 범위는 다음으로 제한한다.

- `4. 고객 발송 본문`
- `5. 고객 작성 양식`
- `6. 파일 제출 안내`
- `7. 보내지 말아야 할 자료`
- `8. 확인과 동의`

가상 리허설 결과:

| 항목 | 판정 | 메모 |
|---|---|---|
| 고객 발송 본문 | `PASS` | 가상값을 넣어도 문장 흐름이 자연스럽다. |
| 고객 작성 양식 | `PASS` | 기본 정보, 수업 정보, 강사진, 상담, 디자인 선호, 공지로 구성되어 있다. |
| 파일 제출 안내 | `PASS` | 로고와 대표 사진 안내가 분리되어 있다. |
| 사진 안내 | `PASS` | 원본 제출 가능, 사용 가능성 확인 문구가 있다. |
| 보내지 말아야 할 자료 | `PASS` | 개인정보, 결제 정보, 시험지 원본, 허위 실적을 차단한다. |
| 확인과 동의 | `PASS` | 공개 가능 정보, 개인정보 미포함, 사진 사용 동의, 게시 전 확인 절차가 있다. |
| 내부 발송 체크 제외 | `PASS` | 고객에게 보낼 범위에서 제외되어 있다. |

## 6. 오해 가능성 점검

| 오해 가능성 | 판정 | 이유 |
|---|---|---|
| 고객이 직접 편집하는 제작툴로 오해 | 낮음 | “직접 편집하는 제작툴이 아니다”가 본문에 있다. |
| 무제한 맞춤 제작으로 오해 | 낮음 | 타 사이트 복제, CSS/코드 수정 요청을 차단한다. |
| 모든 항목을 반드시 채워야 한다고 오해 | 낮음 | “완벽히 채우지 않으셔도 됩니다”가 있다. |
| 사진을 보내면 무조건 게시된다고 오해 | 낮음 | 사용 가능 여부 확인과 NO-GO 기준이 있다. |
| 개인정보를 보내도 된다고 오해 | 낮음 | 보내지 말아야 할 자료와 동의 문구가 반복된다. |
| 샘플 링크가 실제 고객용이라고 오해 | 중간 | localhost 링크는 실제 발송 전 배포 URL로 교체해야 한다. |

## 7. 누락 또는 보완 필요

현재 리허설에서 확인된 보완 필요 항목은 구현 문제가 아니라 운영 준비 항목이다.

1. 실제 고객에게 보낼 때는 localhost 샘플 링크를 쓰면 안 된다.
2. 배포 URL 또는 고객에게 보여줄 미리보기 URL이 필요하다.
3. 베틀 시스템 문의 연락처는 실제 응답 가능한 채널이어야 한다.
4. 파일 제출 방식은 고객별로 하나만 명확히 지정하는 것이 좋다.
5. 실제 고객 정보는 저장소에 남기지 않는 원칙을 유지해야 한다.

## 8. 결론

가상 파일럿 발송 리허설은 통과했다.

현재 패키지는 실제 고객이 생겼을 때 저장소 밖에서 고객별 값을 채워 사용할 수 있는 기본 양식이다. 지금 단계에서 실제 고객 발송 승인이나 고객별 정보 입력은 필요하지 않다.

다음으로는 고객이 회신했다고 가정한 가상 접수 리허설을 최신 발송 문안 기준으로 다시 확인하는 것이 적절하다.
