# 고객 자료 수집 구현 방식 의사결정

Status: Decision  
Project: homepage  
Last Updated: 2026-06-27

---

## 1. 결정

첫 고객 파일럿 전에는 고객 자료 수집 양식 전체 API와 내부 입력 화면을 구현하지 않는다.

첫 파일럿은 문서 양식 기반으로 운영한다. 묵산 내부 제작자가 고객에게 자료 수집 양식을 보내고, 받은 자료를 검수한 뒤 현재 `AcademySite` seed 구조와 `publication.assets` metadata에 수동 반영한다.

현재 구현된 범위는 유지한다.

- 자료 수집 논리 스키마 문서
- `mapIntakeAssetsToPublicationAssets` helper
- 콘텐츠 준비도 점검
- 고객 게시용 logo/hero asset 승인 검증
- `/internal`의 읽기 전용 asset 승인 준비도 표시

---

## 2. 이유

이 프로젝트는 고객에게 홈페이지 제작툴을 판매하는 프로젝트가 아니다. 묵산 내부 제작자가 표준 템플릿 안에서 고객 학원의 홈페이지를 제작하고, 상담, 공지, 결제, 학원관리, 시험지 생성관리로 연결하는 외부 접점 시스템이다.

따라서 첫 파일럿에서 우선 검증해야 할 것은 입력 시스템의 편의성이 아니라 다음이다.

- 고객이 어떤 자료를 실제로 제공하는가
- 어떤 항목에서 누락과 오해가 반복되는가
- 로고, 사진, 강사 경력, 공개 문구의 승인 절차가 현실적으로 작동하는가
- 묵산 1인 또는 극소수 운영자가 수동 제작을 감당할 수 있는가
- `trust-basic-v1` 템플릿이 실제 고객 학원에도 충분한 품질을 내는가

API와 내부 입력 화면을 먼저 만들면 아직 확정되지 않은 양식과 운영 절차가 코드로 굳어진다. 또한 고객 자료 업로드, 파일 권한, 개인정보 차단, 임시 저장, 검수 상태, 수정 이력 같은 부수 범위가 빠르게 늘어난다.

---

## 3. 이번 단계에서 하지 않을 것

첫 파일럿 전에는 다음을 구현하지 않는다.

- 고객용 자료 입력 포털
- 고객 계정, 로그인, 권한
- 파일 업로드 UI
- 자료 수집 CRUD API 전체
- 내부 제작자용 전체 intake 입력 화면
- 고객이 직접 홈페이지 내용을 수정하는 편집 화면
- 자료 제출 후 자동 홈페이지 초안 생성
- 고객별 자유형 섹션 조립

---

## 4. 첫 파일럿 운영 방식

첫 파일럿은 다음 방식으로 진행한다.

1. 베틀 시스템이 문서 양식으로 자료를 요청한다.
2. 고객은 공개 가능한 학원 정보, 디자인 선호, 로고와 사진 사용 동의, 강사 경력 확인 여부를 제출한다.
3. 묵산 내부 제작자가 자료를 검수한다.
4. 부족한 항목은 고객에게 다시 요청한다.
5. 자료가 충분하면 내부 상태를 `MATERIALS_READY`로 전환한다.
6. 베틀 시스템이 `AcademySite` 초안을 구성한다.
7. `content:validate`, 내부 콘텐츠 탭, asset 승인 준비도 패널로 공개 가능성을 점검한다.
8. 고객 확인 후 `CUSTOMER_PUBLISHED` 전환 여부를 판단한다.

---

## 5. API와 내부 입력 화면으로 승격하는 기준

다음 조건 중 일부가 실제로 반복되면 자료 수집 API와 내부 입력 화면 구현을 다시 검토한다.

- 파일럿 고객이 3곳 이상으로 늘어 수동 문서 관리가 병목이 된다.
- 같은 누락 항목이 2회 이상 반복되어 자동 체크가 필요해진다.
- 내부 제작자가 2명 이상이 되어 자료 상태 인수인계가 어려워진다.
- 로고, 사진, 강사 경력, 공지 공개 여부의 승인 이력이 필요해진다.
- 고객 수정 요청이 많아져 문서와 seed 사이 불일치가 자주 생긴다.
- 실제 asset 저장소와 업로드 정책이 결정된다.

승격하더라도 고객용 자유 편집기가 아니라 내부 제작 운영 도구로 시작한다.

---

## 6. 승격 시 최소 구현 범위

향후 구현한다면 1차 범위는 다음으로 제한한다.

- 내부 전용 intake record 생성과 조회
- 필수/권장 자료 누락 체크
- logo/hero asset metadata 입력
- 공개 가능 정보 확인, 개인정보 미포함 확인, 사진 사용 동의 체크
- `AcademySite` 초안으로 수동 변환하기 쉬운 요약 화면

1차 범위에 포함하지 않는다.

- 고객 계정
- 고객 직접 입력 포털
- 파일 업로드 자동화
- 자동 홈페이지 생성
- 레이아웃 편집
- CSS/코드 수정 권한

---

## 7. 현재 결론

이 문서 작성 당시의 1순위는 구현 확대가 아니라 첫 파일럿 운영 자료를 정리하는 것이었다.

문서 양식 기반의 첫 파일럿 자료 요청 패킷은 `docs/14_FIRST_PILOT_CUSTOMER_INTAKE_PACKET.md`에 둔다. 첫 가상 고객 자료 수집 리허설은 `docs/15_FIRST_PILOT_INTAKE_REHEARSAL.md`에 기록한다.

리허설 결과, 첫 파일럿 전에는 전체 API와 내부 입력 화면을 만들 필요가 아직 없다고 판단했다. 당시 다음 작업은 자료 요청 패킷의 문항을 작게 보완해 대표 시간표, 텍스트 로고 fallback, 사진 NO-GO, 강사 경력 공개 방식, 상담 문구 톤에서 생길 혼선을 줄이는 것이었다.
