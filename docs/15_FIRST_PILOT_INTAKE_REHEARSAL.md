## 1. 목적

이 문서는 `docs/14_FIRST_PILOT_CUSTOMER_INTAKE_PACKET.md`를 기준으로 가상 고객 1곳에 대해 자료 수집 리허설을 수행한 기록이다.

목표는 실제 고객 자료를 받기 전에 다음을 확인하는 것이다.

- 고객이 질문에 답할 수 있는가
- 어떤 항목이 비거나 애매해지는가
- `WAITING_FOR_MATERIALS`, `MATERIALS_READY`, `CUSTOMER_PUBLISHED` 판단이 가능한가
- 자료 수집 API나 내부 입력 화면이 지금 필요한가
- 자료 요청 패킷을 보완해야 하는가

이 리허설은 실제 고객 정보나 실제 개인정보를 사용하지 않는다.

## 2. 리허설 시나리오

가상 고객:

| 항목 | 내용 |
|---|---|
| 학원 유형 | 중등, 고등 국어 전문 학원 |
| 규모 | 원장 1명, 강사 1명, 상담은 원장이 직접 처리 |
| 홈페이지 목적 | 상담 문의 접수와 학원 신뢰 형성 |
| 선호 분위기 | 신뢰형, 소수정예형 |
| 자료 준비 상태 | 텍스트 정보는 대부분 준비, 로고와 사진은 일부 부족 |
| 주의사항 | 강사 경력 일부는 사실 확인 전, 사진 일부에 학생 얼굴이 포함됨 |

가상 고객명은 `한빛국어학원`으로 둔다. 실제 학원과 무관한 리허설 데이터다.

## 4. 파일과 asset 검수

| 자료 | 가상 제출 | 판정 | 조치 |
|---|---|---|---|
| 로고 | 별도 파일 없음 | 보완 가능 | 텍스트 로고 fallback 승인 필요 |
| 대표 사진 | 교실 사진 2장 | 일부 보류 | 1장은 학생 얼굴이 있어 사용 불가 |
| 입구 사진 | 없음 | 권장 보강 | 없어도 초안 제작 가능 |
| 상담 공간 사진 | 1장 | 사용 가능 | 사람이 없고 개인정보 노출 없음 |
| 강사 사진 | 원장 사진 1장 | 확인 필요 | 본인 사용 동의 확인 필요 |

asset 기준 판단:

- `publication.assets.logo.assetId`: 없음
- `publication.assets.logo.textFallbackApproved`: 고객 확인 후 `true` 가능
- `publication.assets.hero.assetId`: 상담 공간 사진 또는 교실 사진 중 개인정보 노출 없는 1장 사용 가능
- `publication.assets.hero.approvedForPublish`: 사진 사용 동의 확인 전에는 `false`

현재 상태에서는 `MATERIALS_READY` 전환은 가능하지만, `CUSTOMER_PUBLISHED` 전환은 사진 사용 동의와 텍스트 로고 fallback 승인 후 가능하다.

| 확인 항목 | 가상 응답 | 판정 |
|---|---|---|
| 공개 가능 정보 확인 | 동의 | 통과 |
| 개인정보 미포함 확인 | 동의 | 통과 |
| 로고와 사진 사용 동의 | 일부 동의 | 보완 필요 |
| 초상권과 사용 동의 | 강사 사진 확인 필요 | 보완 필요 |
| 강사 경력 사실 확인 | 경력 문구 보류 | 보완 필요 |

NO-GO:

- 사진 사용 동의가 완료되기 전 `CUSTOMER_PUBLISHED`로 전환하지 않는다.

## 6. 제작 상태 판단

| 상태 | 판단 |
|---|---|
| `WAITING_FOR_MATERIALS` | 첫 회신 전 또는 필수 정보 누락 시 |
| `MATERIALS_READY` | 현재 가상 답변 기준 전환 가능 |
| `DRAFT_CREATED` | 베틀 시스템이 문구 정리와 초안 제작 후 가능 |
| `CUSTOMER_REVIEW` | 내부 검수 후 고객 확인 화면 공유 시 가능 |
| `CUSTOMER_PUBLISHED` | 사진 사용 동의, 텍스트 로고 fallback 승인, 경력 문구 정리 후 가능 |

결론:

현재 가상 고객 답변은 `MATERIALS_READY`로 전환 가능하다. 다만 실제 게시 전에는 asset과 경력 문구 검수가 남는다.

## 7. 누락과 혼선 기록

### 7.1 누락 가능성이 높은 항목

| 항목 | 리허설 결과 | 보완 방향 |
|---|---|---|
| 대표 시간표 | 고객이 요일과 시간만 적을 가능성이 높음 | 과목, 학년, 반 이름 예시를 더 명확히 제공 |
| 로고 | 파일이 없을 수 있음 | 텍스트 로고 fallback 승인 문구를 별도 체크로 둠 |
| 대표 사진 | 개인정보 노출 사진이 섞일 수 있음 | 사람 얼굴, 이름표, 성적표, 연락처 노출 금지 예시 강화 |
| 강사 경력 | 확인 전 경력 문구가 들어올 수 있음 | 경력은 증빙 가능하거나 고객 책임 확인된 경우만 사용한다고 재안내 |
| 공지 공개 여부 | 내부 공지와 공개 공지를 혼동할 수 있음 | 공개 가능한 공지만 작성하라고 강조 |

### 7.2 고객이 헷갈릴 수 있는 문항

| 문항 | 혼선 |
|---|---|
| 상담 유도 문구 톤 | 고객이 구체 문구를 직접 써야 한다고 오해할 수 있음 |
| 참고하고 싶은 샘플 | 고객이 외부 사이트 복제를 요청할 가능성이 있음 |
| 강사 사진 사용 여부 | 사진 제출과 초상권 동의를 같은 의미로 볼 수 있음 |
| 로고 파일 | 로고가 없으면 제작이 불가능하다고 오해할 수 있음 |

### 7.3 패킷 개선 후보

첫 실고객 발송 전 패킷에 다음 보완을 고려한다.

- 대표 시간표 예시를 `요일 / 시간 / 학년 / 과목 / 반 이름` 형식으로 제시
- 로고가 없을 때 `텍스트 로고로 시작해도 됩니다` 체크 항목 추가
- 사진 제출 안내에 `학생 얼굴이 보이면 사용하지 않습니다`를 더 눈에 띄게 표시
- 강사 경력 항목에 `증빙 가능 / 비공개 / 수업 방식만 공개` 선택지를 추가
- 상담 문구 톤에 `베틀 시스템이 문구를 정리합니다` 설명 추가

## 8. API와 내부 입력 화면 필요성 판단

이번 리허설 기준으로는 자료 수집 전체 API와 내부 입력 화면은 아직 필요하지 않다.

이유:

- 가상 고객 1곳 기준으로 문서 양식만으로 `MATERIALS_READY` 판단이 가능하다.
- 누락 항목은 대부분 문항 예시와 안내 문구 보완으로 해결 가능하다.
- 고객 자료 업로드, 임시 저장, 권한, 파일 검수 이력까지 구현하면 범위가 커진다.
- 첫 파일럿에서는 실제 자료의 질과 누락 패턴을 먼저 보는 것이 더 중요하다.

다만 다음 조건이 실제 파일럿에서 확인되면 구현을 재검토한다.

- 사진과 로고 사용 동의 추적이 반복적으로 헷갈림
- 고객 수정 요청이 많아 문서와 seed 사이 불일치가 커짐
- 3곳 이상 고객을 동시에 진행해 자료 상태 인수인계가 어려움
- 파일 업로드와 검수 이력이 필요해짐

## 9. 다음 작업

리허설 결과, 첫 보완 범위는 패킷 문항의 작은 보완이다.

보완 범위:

- 대표 시간표 작성 예시 강화
- 텍스트 로고 fallback 승인 체크 추가
- 사진 제출 NO-GO 문구 강조
- 강사 경력 공개 방식 선택지 추가
- 상담 문구 톤은 베틀 시스템이 정리한다는 안내 추가

이 보완은 고객용 포털이나 API 구현이 아니라 문서 패킷 품질 개선이다.

`docs/14_FIRST_PILOT_CUSTOMER_INTAKE_PACKET.md`에는 위 보완을 반영했다. 다음 작업은 실제 고객에게 보내기 전 묵산이 베틀 시스템 문안 톤과 질문량을 최종 검토하는 것이다.
