# 내부 정적 샘플 자료 상담 리허설 체크리스트

Status: Draft  
Project: homepage  
Last Updated: 2026-06-30

---

## 1. 목적

이 문서는 묵산 내부 검토용 정적 샘플 자료를 실제 고객에게 보여주기 전에, 내부에서 상담 흐름과 기대 관리 문구를 리허설하기 위한 체크리스트다.

현재 목표는 구현이 아니다. `planning/17_INTERNAL_STATIC_SAMPLE_REVIEW_PACK.md`를 기준으로 묵산 내부자가 샘플 방향 설명, 고객 권리 안내, 금지 범위 설명, 자료 누락 확인, GO/HOLD/NO-GO 판정을 일관되게 할 수 있는지 검증한다.

---

## 2. 리허설 전제

리허설은 실제 고객 정보 없이 가상 학원 사례로 진행한다.

필수 전제:

- 실제 개인정보를 사용하지 않는다.
- 실제 고객명, 실제 학원명, 실제 전화번호를 사용하지 않는다.
- 허위 실적, 허위 후기, 허위 강사 경력을 만들지 않는다.
- `/samples`, `/gallery`, 고객 선택 API, 고객 포털 구현은 하지 않는다.
- 리허설은 문서와 현재 `trust-basic-v1` 샘플 화면 확인만으로 진행한다.

사용 자료:

| 자료 | 용도 |
|---|---|
| `planning/17_INTERNAL_STATIC_SAMPLE_REVIEW_PACK.md` | 상담 자료 구성 순서 |
| `planning/16_SAMPLE_CARD_COPY_AND_EXPECTATION_GUIDE.md` | 고객에게 보여줄 문구 |
| `planning/15_SAMPLE_DIRECTION_BRIEFS.md` | 방향별 적합/부적합 기준 |
| `/h/sample-korean-academy` | 실제 구현된 `trust-basic-v1` 화면 확인 |
| `docs/24_SCREENSHOT_REGRESSION_TEST_REPORT.md` | 캡처 기준 참고 |

---

## 3. 역할

리허설은 최소 2역으로 진행한다.

| 역할 | 담당 |
|---|---|
| 상담자 | 고객에게 샘플 방향과 제작 방식을 설명한다 |
| 고객 역할 | 질문, 오해, 요구사항을 제시한다 |
| 기록자 | 가능하면 별도 지정. 없으면 상담자가 리허설 후 기록한다 |

1인 운영 상황에서는 한 사람이 상담자와 기록자를 겸해도 된다. 다만 고객 역할 질문은 미리 정해진 질문 목록을 그대로 사용한다.

## 4. 리허설 시나리오

가상 고객:

```text
학원명: 가상 늘봄국어학원
대상: 중1-고1
과목: 국어, 독서, 문법
강점: 작은 반, 과제 점검, 내신 대비
자료 상태: 로고 없음, 사진 일부 있음, 커리큘럼 초안 있음, 실적 자료 없음
고객 성향: 홈페이지 방향은 고르고 싶지만 직접 편집 기능도 궁금해함
```

이 시나리오는 실제 학원과 무관한 리허설 데이터다.

---

## 5. 리허설 진행 순서

### Step 1. 제작 방식 안내

상담자는 다음을 설명한다.

```text
묵산 홈페이지는 고객이 직접 만드는 자유형 빌더가 아니라, 묵산이 준비한 표준 방향과 템플릿 안에서 내부 제작자가 완성하는 방식입니다.
오늘은 최종 디자인을 확정하는 자리가 아니라, 학원에 맞는 제작 방향을 고르는 단계입니다.
```

PASS:

- 고객이 방향 선택과 최종 확정의 차이를 이해한다.
- 상담자가 “직접 편집”, “즉시 생성”, “원하는 대로” 같은 표현을 쓰지 않는다.

FAIL:

- 고객이 샘플 선택을 즉시 게시나 자동 생성으로 이해한다.
- 상담자가 고객별 무제한 맞춤 가능성을 암시한다.

### Step 2. 권리와 제한 안내

상담자는 고객 권리를 설명한다.

고객이 선택할 수 있는 것:

- 학원명, 로고, 대표 문구
- 대표 색상 선호
- 사진 또는 사진 방향
- 신뢰형, 입시형, 소수정예형 같은 분위기
- 강조 장점
- 상담 문구 톤
- 최종 게시 전 확인과 수정 요청

고객이 선택할 수 없는 것:

- 레이아웃 무제한 변경
- 섹션 자유 조립
- CSS/코드 직접 수정
- 타 사이트 복제
- 검수 없는 즉시 게시

PASS:

- 고객 권리가 부정적으로 들리지 않으면서도 범위가 명확하다.
- 고객이 “제한이 있는 대신 묵산이 완성한다”는 구조를 이해한다.

FAIL:

- 제한 설명이 빠져 고객이 자유형 빌더로 기대한다.
- 고객 권리를 지나치게 축소해 단순 납품물처럼 들린다.

### Step 3. 샘플 방향 3종 설명

상담자는 세 방향을 1분 안에 설명한다.

```text
신뢰형 기본은 가장 안정적인 기본 방향입니다.
내신/입시 관리형은 시험 대비 체계가 분명한 학원에 맞지만, 실적 표현은 확인 자료가 있어야 합니다.
소수정예 관리형은 작은 반과 꾸준한 피드백이 실제 강점인 학원에 맞습니다.
```

PASS:

- `trust-basic-v1`은 실제 샘플 확인 가능으로 설명한다.
- `exam-focus-v1`, `small-group-v1`은 방향 예시 또는 브리프 단계로 설명한다.
- 브리프 단계 방향을 이미 완성된 화면처럼 말하지 않는다.

FAIL:

- 세 방향이 모두 이미 구현된 것처럼 들린다.
- 내신/입시형에서 성과 보장처럼 들리는 문구를 쓴다.

### Step 4. 실제 샘플 확인

확인 경로:

```text
/h/sample-korean-academy
```

확인할 영역:

- 첫 화면의 학원명, 대표 문구, 상담 CTA
- 핵심 강점 3개
- 소개 영역
- 강사진
- 커리큘럼
- 공지
- 오시는 길과 상담 신청
- footer 샘플 고지

PASS:

- 상담자가 샘플이 실제 개인정보 없는 예시임을 설명한다.
- 고객이 신뢰형 기본의 정보 구조를 이해한다.
- 상담 신청 동선이 설명된다.

FAIL:

- 샘플 학원을 실제 고객 사례처럼 오해하게 둔다.
- footer 샘플 고지를 설명하지 않는다.

### Step 5. 고객 질문 대응

고객 역할은 다음 질문을 한다.

1. “이걸 고르면 그대로 만들어지나요?”
2. “이 샘플의 메인에 다른 샘플의 섹션을 섞을 수 있나요?”
3. “우리 학원 색으로 바꿀 수 있나요?”
4. “다른 학원 사이트랑 비슷하게 만들 수 있나요?”
5. “성적 상승 사례나 후기를 넣을 수 있나요?”
6. “선택하면 바로 게시되나요?”

상담자 답변 기준은 `planning/16_SAMPLE_CARD_COPY_AND_EXPECTATION_GUIDE.md`의 FAQ를 따른다.

PASS:

- 모든 답변이 표준 템플릿, 자료 확인, 묵산 검수, 고객 확인 절차 안에 머문다.
- 허위 실적과 외부 사이트 복제를 명확히 거절한다.

FAIL:

- “가능합니다”라고 먼저 말하고 나중에 제한을 붙인다.
- 고객이 원하는 대로 다 해줄 수 있다는 인상을 준다.

### Step 6. 자료 준비 체크

상담자는 가상 고객 자료를 기준으로 누락을 판정한다.

| 항목 | 가상 상태 | 판정 |
|---|---|---|
| 학원명 | 있음 | GO |
| 로고 | 없음 | HOLD, 텍스트 로고 승인 필요 |
| 대표 문구 | 초안 필요 | HOLD |
| 대표 색상 선호 | 없음 | 선택 요청 |
| 사진 | 일부 있음 | 식별 정보 검수 필요 |
| 강사진 소개 | 미정 | HOLD |
| 커리큘럼 | 초안 있음 | 보완 필요 |
| 공지 | 없음 | 선택 |
| 주소/전화/운영 시간 | 미정 | HOLD |
| 실적/후기 | 근거 없음 | 사용 금지 |

PASS:

- 로고, 대표 문구, 주소/전화/운영 시간 누락을 잡는다.
- 실적/후기 근거 없음은 사용 금지로 판정한다.

FAIL:

- 실적/후기 근거 없음에도 문구를 만들어 넣자고 한다.
- 사진 식별 정보 검수를 생략한다.

### Step 7. 내부 판정

리허설 마지막에 상담자는 다음 중 하나로 판정한다.

```text
GO: 제작 착수 가능
HOLD: 추가 자료 필요
NO-GO: 정책 위반 또는 기대 관리 실패
```

가상 시나리오 권장 판정:

```text
HOLD
```

사유:

- 로고 또는 텍스트 로고 승인 필요
- 대표 문구 필요
- 강사진 소개 필요
- 주소/전화/운영 시간 필요
- 실적/후기는 근거 없음으로 제외

## 6. 상담자 체크리스트

리허설 후 상담자는 다음을 체크한다.

| 항목 | PASS | HOLD | FAIL |
|---|---:|---:|---:|
| 제작 방식이 자유형 빌더가 아님을 설명했다 |  |  |  |
| 샘플 선택이 최종 디자인 확정이 아님을 설명했다 |  |  |  |
| 고객 권리와 제한을 함께 설명했다 |  |  |  |
| `trust-basic-v1`만 실제 구현 샘플임을 구분했다 |  |  |  |
| 브리프 단계 방향을 완성 화면처럼 말하지 않았다 |  |  |  |
| 허위 실적/후기/경력 금지를 설명했다 |  |  |  |
| 타 사이트 복제 요구를 거절했다 |  |  |  |
| 고객 자료 누락을 판정했다 |  |  |  |
| GO/HOLD/NO-GO를 기록했다 |  |  |  |
| 다음 요청 자료를 명확히 정리했다 |  |  |  |

FAIL이 하나라도 있으면 실제 고객 상담 전에 문구와 자료 구성을 보완한다.

## 7. 고객 오해 유발 문구 점검

리허설 중 다음 표현이 나오면 즉시 수정한다.

사용 금지:

- “그대로 만들어드립니다”
- “원하는 대로 다 바꿀 수 있습니다”
- “선택하면 바로 제작됩니다”
- “직접 편집하시면 됩니다”
- “다른 사이트처럼 똑같이 가능합니다”
- “후기는 AI로 만들 수 있습니다”
- “실적은 보기 좋게 정리해드리겠습니다”

대체 표현:

- “이 방향을 참고해 학원 자료에 맞게 제작합니다”
- “표준 템플릿 안에서 조정합니다”
- “묵산 내부 제작과 검수 후 고객 확인을 거칩니다”
- “확인 가능한 자료만 공개 문구에 반영합니다”
- “타 사이트 복제가 아니라 묵산 표준 시스템 안에서 새로 제작합니다”

## 9. GO / HOLD / NO-GO 기준

GO:

- 상담자가 제작 방식과 제한을 명확히 설명했다.
- 고객 역할이 샘플 선택을 방향 선택으로 이해했다.
- 자료 누락이 없거나 제작 착수에 지장이 없다.
- 허위 실적, 개인정보, 타 사이트 복제 요구가 없다.

HOLD:

- 고객 기대는 통제됐지만 자료가 부족하다.
- 로고, 대표 문구, 강사진, 주소/전화 등 필수 공개 정보가 부족하다.
- 브리프 단계 방향을 보여줄 때 추가 설명이 필요하다.

NO-GO:

- 고객이 자유형 빌더로 오해한다.
- 상담자가 무제한 맞춤을 암시했다.
- 허위 실적, 허위 후기, 확인되지 않은 강사 경력을 넣으려 한다.
- 타 사이트 복제 요구를 수용했다.
- 검수 없는 즉시 게시처럼 설명했다.

## 10. 이번 체크리스트의 결론

이 체크리스트는 샘플 갤러리 구현 전에 필요한 운영 검증 단계다.

정적 자료만으로 내부 리허설이 통과해야 다음 단계로 넘어갈 수 있다. 리허설에서 고객 오해가 반복되면 갤러리 화면을 구현하기 전에 문구, 카드 구조, 자료 순서를 먼저 수정해야 한다.
